## 10. Coverage Scope Classification

### A. Initial coverage and test-expansion priority

- Pure calculation and transformation logic, including analytics, metrics, scoring, revenue, reports, comparisons, and validation
- State-transition, link, risk, safety, quota, freshness, and eligibility decisions when dependencies can be supplied as inputs
- Serialization, normalization, pagination, scheduling decisions, and other deterministic helpers
- Existing `src/lib/analytics/metrics.ts` behavior already exercised by the current tests

These files remain in the measurement boundary even when they do not yet have tests.

### B. Mock-enabled unit or component scope

- WordPress, Google, X, social, and other external API clients
- Modules using `fetch` or external HTTP communication
- Notification and publishing orchestration
- Time, retry, quota, and provider-response behavior

Tests must use controlled substitutes and must not use live credentials or send data externally.

### C. Integration-test design required

- Prisma/PostgreSQL-dependent services
- Synchronization jobs and locks
- Workers, scheduler, and cron entry points
- Multi-service orchestration and persistence-backed state transitions

These may require isolated databases, dependency injection, or process-boundary tests. Difficulty or initially low coverage is not an exclusion justification.

### D. Technical exclusion candidates

- Generated code
- Type declaration-only files
- Test/spec files and test fixtures/mocks
- Framework-generated directories and coverage output
- A bootstrap entry point only when it contains no independently testable behavior

Each concrete exclusion must be reviewed in the Baseline phase. No production directory is approved for blanket exclusion.

## 11. Include Design

The implementation should explicitly include production `src/**/*.ts` and `src/**/*.tsx` files, subject only to reviewed technical exclusions. An all-files-style boundary is preferred so unimported production modules remain visible.

The precise Vitest glob syntax must be confirmed against the installed provider and repository layout during implementation. This task does not modify `vitest.config.ts`.

## 12. Exclude Design

| Candidate | Technical reason | Re-evaluation condition |
|---|---|---|
| `**/*.{test,spec}.{ts,tsx}` | Test code is not production coverage scope | Repository test-location policy changes |
| `**/__tests__/**` | Test container, not production behavior | Production code is stored there |
| Declaration-only files such as `**/*.d.ts` | No executable behavior | Runtime behavior becomes associated with the path |
| Generated source | Generated implementation is not human-authored production logic | Generation ownership or strategy changes |
| Test fixtures and mocks | Support artifacts distort production coverage | Review each concrete directory before exclusion |
| `coverage/**` and framework build output | Generated measurement/build artifacts | Keep outside source measurement |
| Behavior-free bootstrap/config entry points | No independently testable runtime decision | Behavior is added |

Whole directories must not be excluded because they are untested, integration-heavy, or lower the percentage. Every implemented exclusion requires a documented path, reason, owner, and review trigger.

## 13. Coverage Metrics

- Statements: executed statement coverage
- Branches: alternative-condition coverage and the primary signal for untested decisions
- Functions: invoked callable coverage
- Lines: executed source-line coverage

All four metrics must be reviewed independently. No single aggregate, line-only, or statement-only value is sufficient.

## 14. Baseline Measurement Plan

1. Reconfirm the clean baseline and existing Unit Test result in a separate implementation task.
2. Add one reviewed Vitest coverage provider and compatible configuration.
3. Add `test:coverage` while preserving `test` and the existing Unit Test workflow.
4. Configure explicit production inclusion and only justified technical exclusions.
5. Run locally on Windows without numeric thresholds.
6. Add the separate reporting workflow and run the same command on Ubuntu.
7. Compare source sets, metric summaries, warnings, and exit behavior across platforms.
8. Correct scope or exclusion problems before interpreting percentages.
9. Add priority-domain tests without weakening the measurement boundary.
10. Require at least two stable CI executions before proposing enforcement thresholds.

Candidate reporters are terminal/text summary and machine-readable JSON summary, with LCOV or HTML available as time-limited CI artifacts. `coverage/` must remain untracked and contain no secrets or environment values. No external coverage service is part of initial rollout.

## 15. Threshold Decision Policy

This design intentionally sets no numeric threshold. Thresholds may be proposed only after a valid baseline exists, scope and exclusions are reviewed, platform differences are understood, important gaps are addressed, and at least two stable executions are recorded.

- Conservative: a regression floor based on the stable current result
- Balanced: a floor applied after priority logic receives additional tests
- Target: a medium-term improvement objective, not an unsupported initial gate

The Human Owner must decide separate values for Statements, Branches, Functions, and Lines. Enforcement belongs to a distinct reviewed change after reporting is stable.

## 19. Risks and Limitations

- One test file and four tests cover only a small portion of 104 production files.
- No valid coverage percentage exists yet.
- Provider behavior for transforms, unimported modules, branches, and source maps can vary by version.
- Windows and Ubuntu can differ in path, filesystem, timing, or module discovery behavior.
- Prisma, workers, schedulers, and external clients require deliberate test boundaries and no live-system contact.
- Poor exclusions can create an artificially favorable baseline.
- Premature thresholds can either block routine work or provide false assurance.
- Coverage measures execution, not assertion quality, domain correctness, or security completeness.
